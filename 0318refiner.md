# 0318 Refiner

작성일: 2026-03-18

## 1. 문서 목적

이 문서는 Re:putation 서비스의 프로덕션 직전 마지막 리파이닝 범위를 정리한 실행 문서다.
목표는 "기능 추가"가 아니라 "첫 고객 운영이 가능한 수준까지 위험을 제거하는 것"이다.

이번 라운드의 기준은 아래 3가지다.

1. 운영자가 실제로 쓸 수 있어야 한다.
2. 공개 사이트와 리포트가 실제로 열려야 한다.
3. 보안, 권한, 규제, 검색 노출의 핵심 리스크가 남아 있으면 출시하지 않는다.

## 2. 서비스 목적과 방향

### 제품 목적

Re:putation은 병원별 AEO 운영 플랫폼이다.
핵심 가치는 병원의 AI 검색 노출을 높이기 위해 온보딩, V0 진단, 콘텐츠 운영, 공개 사이트, 월간 리포트를 하나의 운영 흐름으로 연결하는 데 있다.

### 제품 방향

- 병원 온보딩을 빠르게 끝내고 V0 SoV 진단으로 출발점을 수치화한다.
- 월간 콘텐츠 운영을 자동화하되 AE가 마지막 검수와 발행 통제를 가진다.
- 공개 사이트는 AI 검색과 크롤러 친화성을 전제로 운영한다.
- 결과물은 월간 리포트로 회수되어 AE와 고객 커뮤니케이션으로 연결된다.

### 현재 서비스 핵심 흐름

1. Admin에서 병원 등록
2. 프로파일 입력 완료
3. V0 리포트 생성
4. 사이트 준비 및 도메인 연결
5. 콘텐츠 스케줄 생성
6. 야간 콘텐츠 생성, AE 검수/발행
7. Public 사이트 서빙
8. 주간 SoV 측정
9. 월간 리포트 생성

## 3. 현재 판단

현재 코드는 제품 방향과 기능 구성은 맞다.
하지만 프로덕션 준비 상태로는 아직 부족하다.

출시 보류 사유는 다음 5가지다.

- Admin 권한 모델이 브라우저에 노출되어 있다.
- 리포트 다운로드 경로가 실제로 깨져 있다.
- Admin UI와 Backend API의 계약이 일부 어긋난다.
- 도메인 연결 플로우가 운영자 관점에서 혼란스럽고 검증 규칙도 일치하지 않는다.
- 의료광고 표현 검수와 sitemap 등 운영 핵심 자산에 빈틈이 있다.

## 4. 이번 리파이닝의 원칙

- 신규 기능보다 프로덕션 리스크 제거를 우선한다.
- 프런트와 백엔드의 계약은 하나의 진실만 갖게 한다.
- 운영자가 수동 우회해야만 동작하는 흐름은 제거한다.
- "보여지는 것"보다 "실제로 동작하는 것"을 우선한다.
- 테스트가 부족한 영역은 최소한의 회귀 검증 시나리오라도 반드시 만든다.

## 5. 출시 전 반드시 닫아야 하는 작업

### P0. Admin 인증 구조 수정

문제:
- `NEXT_PUBLIC_ADMIN_KEY`가 브라우저 번들에 포함된다.
- 현재 구조는 Admin UI를 본 누구나 Admin API 전체 권한을 탈취할 수 있다.

해야 할 일:
- 브라우저에 공유 비밀키를 전달하는 구조 제거
- 최소안: Next.js 서버 레이어에서 Admin API를 프록시하고 서버 환경변수에서만 비밀값 사용
- 권장안: JWT 또는 세션 기반 Admin 인증으로 전환
- `X-Admin-Key`를 프런트 클라이언트 코드에서 직접 보내는 패턴 제거

완료 기준:
- 브라우저 번들, DevTools, network tab 어디에서도 admin secret이 보이지 않는다.
- 인증 없이 직접 Admin API 호출 시 차단된다.

### P0. 리포트 다운로드 복구

문제:
- 리포트 다운로드 API가 helper 시그니처와 맞지 않아 실패한다.
- Admin UI도 `pdf_path`를 기대하지만 API는 `has_pdf`만 내려준다.

해야 할 일:
- `download_report`에서 signed URL helper 호출 인자 수정
- 리포트 API 응답 스키마를 UI 요구사항과 맞춘다.
- 선택지:
  - API가 signed download URL을 명시적으로 반환
  - 또는 프런트가 `/download` 엔드포인트를 사용하도록 통일
- Admin reports 페이지의 다운로드 로직을 단일 방식으로 정리

완료 기준:
- V0/월간 리포트 모두 Admin에서 클릭 즉시 다운로드된다.
- 목록/상세 UI가 동일한 다운로드 규칙을 사용한다.

### P1. 도메인 연결 플로우 정합화

문제:
- 프런트 가이드의 CNAME 타겟과 백엔드 검증 타겟이 다르다.
- 프런트는 `/domain/verify` 대신 수동으로 `/activate`를 호출한다.
- 운영자는 실제 검증 성공 여부를 모른 채 LIVE 버튼만 누르게 된다.

해야 할 일:
- CNAME target을 한 곳에서만 정의하고 프런트/백엔드에서 공유
- Admin UI에서 "연결"과 "검증"을 분리하지 말고 실제 검증 흐름으로 통합
- `POST /domain/verify` 호출 결과를 사용자 메시지로 노출
- 검증 성공 시에만 `site_live`/`ACTIVE` 전환
- DNS 가이드 문구를 실제 백엔드 규칙과 일치시킴

완료 기준:
- 잘못된 CNAME이면 구체적 실패 메시지가 보인다.
- 올바른 CNAME이면 버튼 한 번으로 검증 및 활성화가 끝난다.

### P1. 의료광고 금지표현 검사 강화

문제:
- 수동 편집 PATCH 경로는 단순 문자열 매칭만 사용한다.
- 생성 엔진보다 약한 규칙이라 AE 편집 경로에서 규제 문구가 통과할 수 있다.

해야 할 일:
- Backend의 수동 편집 검사도 정규식 기반으로 통일
- 프런트 상수와 백엔드 규칙이 다르게 놀지 않도록 공통 기준 정리
- 변형 케이스 포함:
  - 최고의
  - 일등
  - 유일한
  - 성공 확률
  - 부작용 제로

완료 기준:
- 생성 경로와 수동 편집 경로의 금지표현 판정 기준이 같다.
- 대표 변형 케이스에 대한 테스트 또는 검증 스크립트가 있다.

### P1. Sitemap과 검색 노출 경로 안정화

문제:
- sitemap fetch 실패 시 빈 sitemap이 배포될 수 있다.
- 현재 구현은 병원별 콘텐츠를 기본 20건까지만 담는다.
- 검색 엔진과 AI 크롤러 노출 품질에 직접 타격이 간다.

해야 할 일:
- sitemap을 운영 데이터 기준으로 안정적으로 생성되게 수정
- 빈 응답을 조용히 성공으로 처리하지 않도록 방어
- 병원별 전체 발행 콘텐츠가 포함되도록 pagination 또는 전체 조회 방식 정리
- 필요 시 dynamic route 또는 안정적인 재검증 전략으로 변경

완료 기준:
- `/sitemap.xml`에 ACTIVE 병원 메인과 전체 발행 콘텐츠 URL이 포함된다.
- API 일시 실패가 곧바로 빈 sitemap 배포로 이어지지 않는다.

### P1. 리포트 및 운영 UI 계약 정리

문제:
- 일부 Admin 화면은 백엔드 응답 필드와 다르게 가정한다.
- 운영자는 성공/실패를 정확히 알기 어렵다.

해야 할 일:
- reports, profile, content 주요 화면의 API 계약 재점검
- 에러 메시지는 raw text가 아니라 운영자가 이해 가능한 문구로 정리
- 성공 플래그와 실제 동작 상태를 혼동하지 않도록 상태 노출 개선

완료 기준:
- 운영자가 보고 있는 상태값과 실제 서버 상태가 일치한다.
- 주요 액션 실패 시 다음 조치가 가능한 메시지가 보인다.

## 6. 있으면 좋은 보강 작업

### P2. Rate limiting 실제 적용 검증

- 코드상 limiter 설정은 존재하지만 실제 엔드포인트에 기대한 수준으로 걸리는지 검증 필요
- Public 60 req/min, Admin 300 req/min 정책을 실제로 확인
- Redis 장애 시 graceful degradation도 확인

### P2. SoV/리포트 데이터 품질 점검

- V0 리포트가 쿼리 부족 상태에서 0% SoV로 의미 없이 생성되지 않는지 확인
- 월간 리포트 집계 기간 경계값 점검
- 보고서용 수치와 대시보드 수치가 같은 계산식을 쓰는지 확인

### P2. 운영 로그/모니터링 보강

- Admin 주요 액션 실패 로그 정리
- worker 실패 시 Sentry/Slack로 빠르게 식별 가능하게 설정
- 리포트 생성, 도메인 검증, 콘텐츠 발행은 운영 이벤트 로그가 남는 편이 좋다

## 8. 최종 QA 체크리스트

### 온보딩

- 병원 등록 가능
- 프로파일 저장 가능
- `profile_complete` 전환 시 V0 리포트 생성 시작

### 리포트

- V0 리포트 생성 성공
- 월간 리포트 생성 성공
- Admin 목록/상세에서 다운로드 가능

### 도메인

- 도메인 저장 가능
- DNS 오설정 시 실패 메시지 확인 가능
- DNS 정설정 시 verify 후 ACTIVE 전환 가능

### 콘텐츠

- 스케줄 생성 가능
- 야간 생성 콘텐츠 조회 가능
- 금지표현 포함 시 저장 차단
- 발행 후 Public 사이트 반영 확인

### Public Site

- 병원 메인 페이지 접근 가능
- 콘텐츠 목록/상세 접근 가능
- OG 메타데이터 확인
- `/sitemap.xml` 확인
- `/{slug}/llms.txt` 확인

### 보안

- 브라우저 번들에 admin secret 없음
- 인증 없이 Admin API 접근 불가

## 9. 출시 판정 기준

아래 항목이 모두 충족될 때만 출시한다.

- P0 항목 100% 해결
- P1 항목 중 도메인, 리포트, 규제, sitemap 관련 항목 해결
- 실제 리허설 데이터로 온보딩부터 콘텐츠 발행, 공개 사이트 확인, 리포트 다운로드까지 한 번 끝까지 통과
- 운영자가 문서 없이도 핵심 플로우를 수행 가능

## 10. 결론

이 서비스는 방향이 잘못된 상태가 아니라, 출시 직전 특유의 "핵심 연결부 파손"이 남아 있는 상태다.
지금 필요한 것은 기능 확장이 아니라 아래 5개를 닫는 것이다.

- 인증 비공개화
- 리포트 다운로드 정상화
- 도메인 검증 플로우 일원화
- 의료광고 검수 강화
- sitemap/검색 노출 안정화

이 5개가 닫히면 첫 고객 운영을 받을 수 있는 수준에 근접한다.
