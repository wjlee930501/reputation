### AUTH-1 [HIGH] TRUSTED_PROXY_IPS unset in production → rate-limit keys by Google front-end hop, not real client
- **Area:** Auth &amp; access control · **Effort:** S · **Confidence:** 0.82
- **Files:** `backend/app/core/rate_limit.py:14-47`; `backend/app/core/config.py:42`; `terraform/cloudrun.tf:18-65`; `terraform/loadbalancer.tf:41-65`
- **Evidence:** get_request_ip() only honors X-Forwarded-For/X-Real-IP when request.client.host is in settings.TRUSTED_PROXY_IPS (rate_limit.py:17,31-47). The default is ['127.0.0.1','::1'] (config.py:42) and grep of terraform/ shows TRUSTED_PROXY_IPS is NOT set anywhere (confirmed: 'NEITHER ALLOWED_ORIGINS NOR TRUSTED_PROXY_IPS FOUND IN TERRAFORM'). On Cloud Run behind the Serverless-NEG load balancer (loadbalancer.tf:14-22,41-65), request.client.host is a Google internal/GFE address, never 127.0.0.1, so _is_trusted_proxy() returns False and X-Forwarded-For is ignored. The public lead limiter (leads.py:52) and admin limiter (security.py:32) therefore key on the GFE hop IP, not the actual client.
- **Impact:** The public lead form rate limit (5/min;30/hour;100/day) and admin 100/min limit are keyed on a small set of Google front-end IPs shared across all clients. Either every real client collapses into one shared bucket (legitimate users get 429'd / DoS each other) or, where the GFE IP rotates, an attacker rotates across buckets and largely defeats the limit. The anti-bot/anti-DB-flood protection the readiness note credits (line 22) is effectively neutralized in prod.
- **Fix:** Set TRUSTED_PROXY_IPS to the GCP load-balancer/GFE source ranges (or '0.0.0.0/0,::/0' specifically because the only ingress is INTERNAL_LOAD_BALANCER so the immediate hop is always trusted) as an explicit env in terraform/cloudrun.tf for the API service, and document it in .env.production.example. Add a deploy-time assertion that TRUSTED_PROXY_IPS is non-localhost when APP_ENV=production.
- **Existing mitigation:** The proxy-aware logic itself is correctly written and validates the candidate IP; the gap is purely the missing production env value. Cloud Run ingress is INTERNAL_LOAD_BALANCER (cloudrun.tf:10), so the immediate hop is always the trusted LB.
- **Verifier:** confirmed — Mechanism independently verified end-to-end. (1) get_request_ip() (backend/app/core/rate_limit.py:14-28) only honors X-Forwarded-For/X-Real-IP when request.client.host passes _is_trusted_proxy(), which checks membership in settings.TRUSTED_PROXY_IPS (lines 17, 31-47). (2) Default is ["127.0.0.1","::1"] (config.py:42). (3) Confirmed TRUSTED_PROXY_IPS is set NOWHERE: grep of terraform/ returns nothing; the API Cloud Run env block (cloudrun.tf:18-65) lists only SERVICE/APP_ENV/GCP_PROJECT_ID/DB_*/REDIS_URL/GCP_STORAGE_BUCKET/GCS_REPORTS_BUCKET/SITE_REVALIDATE_URL plus local.app_secret_env, and app_secret_env (secretmanager.tf:70-81) contains only 8 named secrets, none of which is TRUSTED_PROXY_IPS. So in prod it falls back to the localhost default. (4) Cloud Run ingress is INGRESS_TRAFFIC_INTERNAL_LOAD_BALANCER (cloudrun.tf:10) behind a SERVERLESS NEG (loadbalancer.tf:14-22), so the TCP peer the container sees is a Google internal serving-proxy IP, never 127.0.0.1.

I specifically checked for the mitigation the auditor might have missed — a uvicorn/Starlette proxy-headers layer that would rewrite request.client.host from XFF before the app sees it. There is none: docker-entrypoint.sh:8 runs `uvicorn app.main:app --host 0.0.0.0 --port ...` with NO --forwarded-allow-ips and NO --proxy-headers override, and main.py adds no ProxyHeadersMiddleware/TrustedHost. uvicorn's built-in ProxyHeadersMiddleware defaults to forwarded_allow_ips="127.0.0.1", so it also refuses to trust the GFE peer and leaves request.client.host as the Google internal hop. Net: _is_trusted_proxy() returns False, XFF is dropped, and both limiters key on the shared Google hop. Both limiters confirmed live: leads.py:52 (@limiter.limit(PUBLIC_LEAD_RATE_LIMIT="5/minute;30/hour;100/day")) and security.py:14,32 (admin 100/minute keyed on f"admin:{get_request_ip(request)}").

Impact assessment (skeptical): behind a Serverless NEG the container peer comes from a small shared pool of Google internal proxies, so the dominant real effect is bucket COLLAPSE — distinct legitimate clients share a handful of keys. (a) The public lead form (5/min) will 429 innocent users under modest concurrent traffic (self-DoS). (b) The explicit P0 control the readiness note line 22 credits ("봇 1대로 DB·Slack 도배 시나리오 차단") is neutralized: one attacker IP is indistinguishable from the aggregate, so per-IP throttling no longer isolates the attacker. (c) Secondary: get_request_ip() is also persisted as consent_ip on every lead (leads.py:71) for PIPA consent records — those legal audit IPs will all be a garbage Google internal address, corrupting the consent trail. No existing mitigation: the proxy-aware code itself is correct and validates the candidate IP; the gap is purely the missing production env value. Severity high is warranted — it defeats a documented P0 security control and corrupts a compliance field — though I note the day-one symptom is shared-bucket collapse/false-positive throttling rather than a clean unlimited bypass, and it only bites under real load/attack; that keeps it at high rather than critical. Fix is small (S): set TRUSTED_PROXY_IPS to the LB/GFE source ranges (or, since ingress is strictly INTERNAL_LOAD_BALANCER, the immediate hop is always the trusted LB so a broad CIDR is acceptable here) as an explicit API env in cloudrun.tf, document in .env.production.example, and add a startup assertion that TRUSTED_PROXY_IPS is non-localhost when APP_ENV=production. Confidence ~0.85; the small residual is GCP-specific behavior of exactly which internal IP/range the container observes, which affects fix wording but not the confirmed broken state.

### AUTH-2 [HIGH] SlowAPIMiddleware not mounted → default rate limits inert; unauthenticated public site endpoints have no rate limit
- **Area:** Auth &amp; access control · **Effort:** S · **Confidence:** 0.85
- **Files:** `backend/app/main.py:49-50`; `backend/app/core/rate_limit.py:64-68`; `backend/app/api/public/site.py:26-123`

The 60/min default the author likely believed was active provides no protection.
- **Fix:** Either mount SlowAPIMiddleware (from slowapi.middleware import SlowAPIMiddleware; app.add_middleware(SlowAPIMiddleware)) so default_limits take effect, or add explicit @limiter.limit decorators to the public site.py read endpoints. Also configure a Cloud Armor rate-based ban rule instead of the current default-allow-all (loadbalancer.tf:136-146) as defense in depth.
- **Existing mitigation:** PublicApiCacheMiddleware (main.py:79-100) sets CDN cache headers (max-age 60-300s) which can absorb repeated identical GETs at a CDN, but enable_cdn=false on the backend service (loadbalancer.tf:51) means no GCP CDN is fronting it, so cache headers only help downstream browsers/ISR, not origin protection.
- **Verifier:** confirmed — Verified every mechanical claim against source. (1) SlowAPIMiddleware is never imported or mounted — grep across the repo returns zero hits; the only app.add_middleware calls in main.py:64-108 are SecurityHeaders, RequestId, PublicApiCache, and CORS. (2) limiter has default_limits=["60/minute"] (rate_limit.py:64-68), but slowapi enforces default_limits ONLY via SlowAPIMiddleware, so with the middleware absent these defaults apply to nothing — a genuine dead/false-sense-of-protection config. (3) public/site.py read endpoints (list_hospitals:26, get_hospital_public:42, get_public_hospital_asset:68, list_published_contents:92, get_content_public:115) carry zero @limiter decorators; each runs uncached DB queries. (4) The one decorated public endpoint, leads.py:52 @limiter.limit, IS protected because decorator-based limits run in the route wrapper independent of the middleware — this is exactly why the author may have wrongly assumed defaults were also active. (5) Admin is independently protected via a manual strategy.hit() in security.py:26-34 (100/min), so admin abuse is bounded. (6) Cloud Armor policy is default-allow-all with no rate-based-ban rule (loadbalancer.tf:132-147), enable_cdn=false (line 51), DB_POOL_SIZE=10 (config.py:106). Net: the public site read endpoints are unthrottled at the application layer (inert defaults, no decorators), at the edge (no Cloud Armor rate rule), and have no CDN absorption — a single unauthenticated internet client can saturate the 10-connection pool and degrade both the public site and admin operations sharing the same Cloud SQL instance. The auditor already correctly identified the PublicApiCacheMiddleware/enable_cdn=false caveat, so no mitigation was missed. Severity stays high: it is unauthenticated, internet-reachable, trivially triggered, and impacts shared availability. The only reason it is not critical is that impact is availability-only (no data exposure or integrity loss — endpoints serve already-public data) and requires deliberate flooding. Effort S is reasonable: either app.add_middleware(SlowAPIMiddleware) or add explicit @limiter.limit decorators to the five site read endpoints, plus a Cloud Armor rate-based-ban rule as defense in depth.
